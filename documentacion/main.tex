\section{Informe de guía práctica}

\subsection{Objetivos}

\subsubsection{General}
Desarrollar un sistema web de facturación para una ferretería, utilizando React, ASP.NET Core, SQL Server y Entity Framework Core, bajo una arquitectura orientada a microservicios con dos bases de datos separadas para clientes y procesos.
\subsubsection{Específicos}
\begin{itemize}[leftmargin=1.2cm]
    \item Transformar los modelos analizados en estructuras físicas de base de datos, definiendo tablas, claves primarias y claves foráneas.
    \item Ingresar registros de prueba en cada esquema para verificar el funcionamiento de las relaciones establecidas.
    \item Ejecutar consultas DQL y subconsultas en Oracle para recuperar información y validar la consistencia de los datos almacenados.
\end{itemize}

\subsection{Instrucciones}
La actividad consiste en desarrollar la implementación física de varios ejercicios de bases de datos en Oracle. Para ello, primero se construyó la estructura de tablas correspondiente a cada caso, considerando claves primarias, claves foráneas y restricciones básicas de integridad. Después se ingresaron datos de prueba con sentencias \texttt{INSERT INTO}, con el propósito de poblar las tablas y disponer de información suficiente para la validación. Finalmente, se elaboraron consultas DQL y subconsultas orientadas a recuperar datos específicos, comprobar las relaciones entre tablas y evidenciar el correcto funcionamiento del esquema físico propuesto.

\subsection{Conclusiones}
El desarrollo de la guía permitió comprender de manera práctica la implementación del modelo físico de una base de datos en Oracle, evidenciando que la creación adecuada de tablas, claves y restricciones constituye la base para el almacenamiento organizado de la información. Asimismo, el ingreso de datos de prueba permitió verificar la validez de las relaciones establecidas y comprobar el funcionamiento de la integridad referencial en los distintos ejercicios.

De igual manera, la ejecución de consultas DQL y subconsultas demostró que una base de datos bien estructurada no solo permite guardar información, sino también recuperarla de forma precisa según diferentes criterios. En consecuencia, la práctica fortaleció la relación entre diseño, implementación y consulta de datos dentro de un entorno real de trabajo con Oracle.

\subsection{Recomendaciones}
Se recomienda verificar cuidadosamente la coherencia entre el modelo relacional y el modelo físico antes de crear las tablas en Oracle, con el fin de evitar errores en claves, nombres de campos o relaciones entre entidades. También es conveniente ingresar datos de prueba que representen distintos escenarios, ya que ello facilita la comprobación del correcto funcionamiento de las restricciones y relaciones definidas.

Además, se recomienda continuar practicando consultas DQL y subconsultas sobre las bases de datos implementadas, puesto que esta fase permite interpretar la información almacenada y validar de manera más completa el trabajo realizado en la construcción del esquema físico.
